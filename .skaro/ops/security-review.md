# Обзор безопасности

## Текущее состояние
- Только прототипный уровень.
- Опциональная HMAC-подпись существует, но транспорт использует обычный HTTP, а доверие на входе очень мягкое.
- Секреты из конфигов живут в YAML и должны оставаться вне коммитов и вне версионируемых Skaro secrets files.

## Основные риски
- Gossip-пиры считаются доверенными только по адресу.
- Нет TLS, mutual auth, политики защиты от replay и rate limiting.
- Demo-визуализатор и endpoint эмиссии могут напрямую создавать runtime-события.

## Минимальные правила на будущее
- Нельзя описывать текущий транспорт как безопасный.
- Если добавляются security-возможности, необходимо явно определить:
  - модель идентичности
  - поведение при ошибке верификации
  - схему ротации ключей
  - шаги настройки для оператора
