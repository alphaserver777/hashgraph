# Профиль памяти MDRJ-DAG: почему K_рес у A4 нелинейный

Документ для разделов 3 и 4 диссертации: эмпирическое обоснование
того, что K_рес распределённого hashgraph-реестра (A4) принципиально
выше, чем у централизованного SIEM (A1), и какие архитектурные слои
этого реестра удерживают рост K_рес ограниченным во времени.

## Наблюдение на стенде

При накоплении ~3000 событий в SQLite на узле с `MemoryMax=512M`
процесс попадает в OOM-restart loop (счётчик systemd доходит до 78
за несколько часов). Это происходит на узлах с историей предыдущих
прогонов (Germany, Germany2), но не на свежих (Zomro, dev1-robots)
— те держатся на 100–140 MB при том же лимите. Отсюда вопрос для
защиты: **почему распределённое решение, написанное как замена
клиент-серверу, оказывается более требовательным к памяти?**

Ответ — это не баг реализации, а **архитектурное свойство BFT-
консенсуса hashgraph**.

## Где растёт память: семь точек диагноза

## Сравнение с архитектурой A1 (классический SIEM)

| Характеристика | A1 (SIEM standalone) | A4 (MDRJ-DAG) |
|---|---|---|
| Хранилище | append-only log на диске | DAG + SQLite + кеш предков в Python |
| Чтение для проверки | Не нужно при онлайн-режиме | Каждый persist пересчитывает всю DAG |
| K_рес/узел (RSS) | O(1) от потока, ~50–80 MB | O(N) от истории + всплески O(R²·M²) |
| K_рес/кластер | один сервер на 1× | N узлов на K_рес каждый |
| P_сохр при отказе сервера | 0 (полная потеря) | ≥(N−⌊N/3⌋)/N |
| Tamper-evidence (УБИ.124) | нет (логи можно подменить) | есть (Merkle + 2/3 кворум) |

**Вывод для главы 3:** A4 жертвует K_рес/узел ради двух свойств,
которых у A1 нет в принципе — tamper-evidence и P_сохр > 0 при
произвольных отказах одной ноды. Без этих свойств защита от
УБИ.124 невозможна, а это — целевая угроза работы.

## Архитектурные слои митигации K_рес

Чтобы K_рес/узел не рос линейно во времени, в A4 реализованы три
слоя сдерживания:

### Слой 1 — `MemoryMax=2G` (нижняя граница)
Файл: [deploy/systemd/mdrj.service](../../deploy/systemd/mdrj.service)

Измерено: пик recompute при N=3000 событий и 4 пирах = ~600 MB.
Лимит установлен с запасом 3×. Без этого слоя — OOM-restart loop.

### Слой 2 — retention под checkpoint
Файлы: [mdrj/node.py:1282](../../mdrj/node.py#L1282) `_retention_loop`,
[mdrj/storage.py](../../mdrj/storage.py) `prune_under_checkpoint`.

При `retention.enabled: true` раз в 5 минут события класса B/C под
подтверждённым merkle-checkpoint удаляются. События класса A
сохраняются всегда. Удалённые события заменяются скелетом хешей
в таблице `event_skeletons` — целостность causal chain
доказуема без хранения payload.

Это **главный архитектурный аргумент для защиты**: K_рес у A4 не
монотонно растёт со временем, а ограничен сверху размером окна
retention. При max_age_days=7 и потоке 50 событий/час получаем
≤ 8400 событий в horizon хранилище — это укладывается в 2G лимит
с большим запасом.

### Слой 3 — дебаунс пересчёта консенсуса
Файлы: [mdrj/node.py:_request_recompute](../../mdrj/node.py),
[mdrj/config.py:RuntimeConfig](../../mdrj/config.py).

При `runtime.recompute_debounce_sec > 0` (по умолчанию 0.3 на стенде)
K последовательных persist коалесцируются в один пересчёт через
`asyncio.to_thread`. Цена: total_order отстаёт на window_sec. Выигрыш:
на gossip-burst в 10 событий за 200мс получаем 1 пересчёт вместо
10 → экономия CPU и пика RSS в десятки раз.

## Эмпирическая верификация

Метрики стенда после применения трёх слоёв:

```
mdrj_rss_bytes (steady state, idle)        < 100 MB
mdrj_rss_bytes (peak under gossip burst)   < 400 MB
mdrj_db_size_bytes (после retention cycle) стабилизируется
mdrj_checkpoint_confirmed_total            растёт по N/checkpoint_window
mdrj_service_killed_total                  = 0 (нет OOM)
```

PromQL-запрос для калибровки K_рес в `simulation.html`:

```promql
# K_рес на узле = max(RSS) / R_max(физический бюджет узла)
max_over_time(mdrj_rss_bytes[24h]) / (2 * 1024 * 1024 * 1024)
```

При R_max=2G и измеренном RSS ≤ 400M получаем K_рес ≤ 0.2 — это
число можно прямо подставить в форму симулятора A4 как эмпирическую
калибровку.

## Что не сделано в рамках прототипа (отложено)

- **Memoization в `_fame_vote_history_and_decisions`**: O(R²·M²) можно
  свести к O(R·M) кешем «votes by witness». Это инвазивно — переписывает
  ядро консенсуса (WIP-задача 014 пользователя). Не делалось в рамках
  данной работы, чтобы не ломать существующую корректность.
- **Backpressure-очередь на gossip-вход**: `asyncio.Queue(maxsize=N)`
  при перегрузке отвечает 503 на `/event/batch`. Архитектурно
  логичный следующий шаг; в прототипе пока не реализован, потому что
  трёх слоёв выше достаточно для стенда из 4 узлов с типичной нагрузкой.

Эти два направления — естественный задел для будущей работы и
явно фиксируются в разделе «Перспективы развития».
