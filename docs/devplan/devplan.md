# Среднесрочный План Развития

## Цель
Сделать репозиторий пригодным для длительной разработки через LLM и людей без расползания архитектуры, без неявных контрактов и без скрытых временных решений.

## Этап 1. Нормализовать Контракты И Документацию
Статус: активный

### 1.1. Закрепить рабочий SDD-процесс в `docs/`
Статус: выполнено

### 1.2. Формализовать API и storage contracts
Статус: следующий приоритет
Причина:
- сейчас поведение распределено между кодом, README и тестами;
- schema change и API change без отдельного contract layer слишком рискованны.

### 1.3. Явно отделить demo/deploy-контур от production-ожиданий
Статус: в очереди
Причина:
- в репозитории есть Docker Compose и demo-кластер;
- подтверждённого production-контура нет, но документы должны явно это фиксировать.

### 1.4. Зафиксировать production-контур Linux sidecar-агента
Статус: следующий шаг
Причина:
- текущие события создаются кодом прототипа и demo-сценариями;
- для реального внедрения нужен отдельный ingestion-layer;
- нужно заранее определить минимальный production-каталог событий доступа и привилегий, прежде чем писать агент.

## Этап 2. Снизить Архитектурные Риски
Статус: запланировано

### 2.1. Проработать policy для schema change и миграций
Фокус:
- SQLite schema evolution;
- backup/rollback;
- совместимость данных между версиями.

### 2.2. Разобрать связанность `mdrj/api.py`
Фокус:
- отделение встроенной визуализации и demo-логики от backend-ядра;
- снижение стоимости сопровождения API-модуля.

### 2.3. Уточнить модель доверия и transport hardening
Фокус:
- HMAC semantics;
- TLS / auth expectations;
- явная граница между demo и production.

### 2.4. Построить минимальный production ingestion для Linux
Фокус:
- container sidecar-агент;
- `journald`, `auth.log`, `sudo`;
- нормализованный контракт события;
- публикация `A/B` событий в локальный узел реестра.

## Этап 3. Сделать Разработку Повторяемой
Статус: запланировано

### 3.1. Поддерживать задачи как основной вход в изменения
Фокус:
- каждая нетривиальная работа начинается с task spec;
- follow-up не теряются, а оформляются новыми задачами.

### 3.2. Ввести ADR для нетривиальных архитектурных решений
Фокус:
- изменения в consensus semantics;
- storage/API breaking changes;
- deploy/security decisions.

### 3.3. Поддерживать документацию как часть Definition of Done
Фокус:
- verification;
- обновление архитектурных и эксплуатационных документов;
- проверка отсутствия скрытых заглушек и неописанных допущений.
