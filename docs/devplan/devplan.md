# Среднесрочный План Развития

## Цель
Сделать репозиторий пригодным для длительной разработки через LLM и людей без расползания архитектуры, без неявных контрактов и без скрытых временных решений.

## Этап 1. Нормализовать Контракты И Документацию
Статус: активный

### 1.1. Закрепить рабочий SDD-процесс в `docs/`
Статус: выполнено

### 1.2. Формализовать API и storage contracts
Статус: следующий приоритет
Причина:
- сейчас поведение распределено между кодом, README и тестами;
- schema change и API change без отдельного contract layer слишком рискованны.

### 1.3. Явно отделить demo/deploy-контур от production-ожиданий
Статус: в очереди
Причина:
- в репозитории есть Docker Compose и demo-кластер;
- подтверждённого production-контура нет, но документы должны явно это фиксировать.

### 1.4. Зафиксировать production-контур самодостаточного Linux-контейнера узла
Статус: выполнено
Причина:
- текущие события создаются кодом прототипа и demo-сценариями;
- для реального внедрения нужен отдельный ingestion-layer;
- нужно заранее определить минимальный production-каталог событий доступа и привилегий, прежде чем писать агент.

### 1.5. Построить первый вертикальный срез Linux ingestion
Статус: активный
Причина:
- после проектирования появился минимальный работающий каркас;
- уже реализован один реальный сигнал `admin_ssh_login_success` через file-based ingestion;
- этап ещё активен, потому что ручной Docker verification и следующий набор Linux-сигналов не закрыты.

### 1.6. Проверить внешний двухузловой Linux-стенд
Статус: активный
Причина:
- локальный bootstrap-контур уже есть;
- теперь нужно подтвердить реальную работу `linux-node` на двух внешних Linux-хостах;
- этот шаг должен доказать, что универсальный контейнер и env-конфиг достаточны для внешнего испытательного стенда.

### 1.7. Зафиксировать единый временной режим внешних узлов
Статус: активный
Причина:
- после появления внешнего стенда нужен единый операционный режим времени;
- для ручной проверки логов, UI и реестра внешние узлы должны использовать NTP и `Europe/Moscow`.

## Этап 2. Снизить Архитектурные Риски
Статус: запланировано

### 2.1. Проработать policy для schema change и миграций
Фокус:
- SQLite schema evolution;
- backup/rollback;
- совместимость данных между версиями.

### 2.2. Разобрать связанность `mdrj/api.py`
Фокус:
- отделение встроенной визуализации и demo-логики от backend-ядра;
- снижение стоимости сопровождения API-модуля.

### 2.3. Уточнить модель доверия и transport hardening
Фокус:
- HMAC semantics;
- TLS / auth expectations;
- явная граница между demo и production.

### 2.4. Построить минимальный production ingestion для Linux
Фокус:
- самодостаточный контейнер узла;
- расширение от первого file-based сигнала к `journald`, `auth.log`, `sudo`;
- нормализованный контракт события;
- публикация `A/B` событий во внутренний runtime локального узла реестра.

### 2.5. Ввести управляемый реестр участников сети
Фокус:
- operator-managed страница участников сети в `/viz`;
- локальное сохранение peer-registry в SQLite;
- исключение скомпрометированного узла из gossip-контура без ручного редактирования конфигов;
- подготовка к следующему шагу discovery/seed-модели.

### 2.6. Сблизить порядок событий с Hashgraph-подобным consensus
Статус: активный
Фокус:
- честно зафиксировать разрыв между текущим `consensus_ts` и полноценным hashgraph-consensus;
- ввести и закрепить первый рабочий этап с `rounds`, `round_received`, membership snapshot и новым reference-порядком;
- расширить explicit fame decision pipeline persisted vote trace-полями `fame_vote_round/yes/no`;
- убрать direct visibility fallback из поздних fame-rounds;
- выделить explicit round-by-round internal vote history для fame pipeline;
- выделить formal fame decision layer поверх этой vote history;
- требовать fully decided fame rounds перед использованием их в `round_received`;
- явно маркировать unresolved fame как `needs_coin`, когда обычные vote rounds исчерпаны;
- ввести deterministic coin surrogate как временный bridge-step для unresolved fame;
- ввести явный `fame_decision_kind`, чтобы downstream различал vote-решение и surrogate bridge-step;
- следующим шагом перейти от fame voting без visibility fallback к full famous witness voting и `virtual voting`;
- усилить integration verification для одновременных событий, плохих часов и partition/recovery;
- покрыть это сценариями одновременных событий, плохих часов и partition/recovery.
