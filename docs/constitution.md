# Конституция Проекта

## Назначение
Этот документ фиксирует инженерные правила для `hashgraph` как репозитория прототипа MDRJ-DAG. Он описывает не идеальную будущую систему, а обязательные правила для текущего кода и дальнейшей разработки через task specs.

## Source Of Truth
- Для фактического runtime-поведения source of truth это код в `mdrj/` и покрывающие его тесты в `tests/`.
- Для классификации известных типов событий source of truth это `docs/event-classification.md`.
  `mdrj/event_catalog.py` обязан быть исполняемым зеркалом этого документа.
- Для текущей архитектурной картины source of truth это `docs/architecture/architecture.md`.
- Для системных инвариантов source of truth это `docs/architecture/invariants.md`.
- Для процесса разработки source of truth это `docs/WORKFLOW.md` и `docs/tasks/PROCESS.md`.
- Для конкретного изменения source of truth это соответствующий task spec в `docs/tasks/`.

Если между документом и кодом есть расхождение, нельзя молча "додумывать". Расхождение должно быть либо исправлено, либо явно зафиксировано в задаче.

## Правила По Env И Конфигурации
- Конфигурация узла задаётся YAML-файлом и загружается через `mdrj/config.py`.
- Секреты нельзя переносить в код, тестовые фикстуры или документацию как норму эксплуатации.
- `security.hmac_key` в коммитнутых demo-конфигах считается только demo-значением, не production-секретом.
- Любые новые env-переменные, секреты или обязательные конфиги должны документироваться в `docs/ops/DEPLOYMENT.md` и, при необходимости, в task spec.
- Нельзя делать скрытую зависимость от локального окружения разработчика без явной документации.
- Для внешних узлов проекта системные часы должны быть синхронизированы через NTP.
- Для эксплуатационного контура и чтения журналов на внешнем стенде используется локальная временная зона `Europe/Moscow`, если отдельной задачей не зафиксировано иное.

## Правила По API
- HTTP API в `mdrj/api.py` и CLI в `mdrj/cli.py` считаются внешними контрактами репозитория.
- Breaking changes в endpoint, payload, semantics, кодах ответа или CLI-потоках допустимы только через отдельную задачу с явным scope.
- Изменения API должны сопровождаться:
  - обновлением task spec;
  - обновлением `docs/architecture/architecture.md` при изменении фактических границ системы;
  - обновлением `README.md` и/или отдельных reference docs, если меняется публичный способ использования.

## Правила По Business Logic
- Source of truth для доменной логики репликации, хранения и упорядочивания находится в `mdrj/node.py`, `mdrj/storage.py`, `mdrj/gossip.py`, `mdrj/prioritization.py`, `mdrj/consensus.py`, `mdrj/vectorclock.py`.
- Нельзя дублировать одну и ту же policy-логику в нескольких местах без явно выделенного источника правды.
- Если появляется новая policy-матрица, справочник или классификация, она должна иметь один явный source of truth: либо документ, либо runtime-модуль, с прямым указанием зеркал.

## Schema Change / Rollback
- SQLite schema сейчас создаётся напрямую в `mdrj/storage.py`; отдельного migration framework в проекте пока нет.
- Любое изменение схемы, структуры записей, индексов или semantics хранения требует отдельной задачи.
- Задача на schema change обязана содержать:
  - описание текущего состояния;
  - план совместимости или осознанный breaking change;
  - verification на старом и новом поведении, если применимо;
  - rollback strategy.
- Пока нет автоматических миграций, rollback для schema/storage changes должен рассматриваться консервативно:
  - через backup/замену SQLite файла или volume;
  - через возврат на предыдущий commit;
  - через явное описание, если rollback без потери данных невозможен.

## Verification Обязательна
- Нельзя считать задачу завершённой без verification.
- Минимальная verification должна соответствовать затронутой области:
  - unit/integration tests для логики и контрактов;
  - ручная проверка CLI/API/deploy-потока, если задача меняет эксплуатационное поведение;
  - проверка документации на согласованность с кодом.
- Если verification не выполнена, это должно быть явно указано в задаче и в результате работы.

## Hidden Stubs И Временные Хаки
- Скрытые заглушки, временные хаки и "потом доделаем" недопустимы как незадокументированное состояние.
- Если временное решение действительно нужно, оно обязано быть:
  - ограничено scope конкретной задачи;
  - явно описано в `Заметки` и `Rollback / Safety`;
  - вынесено в follow-up task spec, если долг остаётся после завершения задачи.

## Изменение Архитектуры И Документации
- Изменение фактической архитектуры требует обновления `docs/architecture/architecture.md`.
- Изменение системных инвариантов требует обновления `docs/architecture/invariants.md`.
- Изменение инженерных правил требует обновления `docs/constitution.md`.
- Изменение deploy/runbook требует обновления `docs/ops/DEPLOYMENT.md`.
- Перед новой задачей и перед реализацией обязательно перечитывать ключевые документы, перечисленные в `docs/WORKFLOW.md`.
