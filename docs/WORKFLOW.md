# Workflow Документации И Разработки

## Зачем нужен этот набор документов
Этот репозиторий уже вышел за пределы "маленького скрипта": здесь есть runtime-узел, HTTP API, SQLite storage, gossip-репликация, demo-визуализация, локальные сценарии и Docker Compose кластер. Без фиксированного spec-driven workflow следующая LLM или разработчик быстро начнут:
- менять поведение без явной задачи;
- дублировать логику и договорённости между кодом, README и ad-hoc заметками;
- вносить скрытые хаки вместо оформленных решений;
- ломать инварианты хранения, API или deploy-потока без нормальной verification.

Эта папка `docs/` нужна как рабочая система документов для долгой разработки через task specs, acceptance criteria, verification и обязательное обновление архитектурных артефактов.

`docs/` является единственной рабочей точкой правды для документации проекта. Материалы в `.skaro/` считаются архивными и не должны использоваться как основной источник текущих правил, процесса или архитектурных решений.

## Основные документы
- `docs/WORKFLOW.md`
  Главный процесс работы по задачам и правила чтения документов.
- `docs/constitution.md`
  Инженерные правила проекта и обязательные ограничения.
- `docs/architecture/architecture.md`
  Текущее состояние архитектуры, реальные source of truth и ограничения.
- `docs/architecture/invariants.md`
  Инварианты системы, которые нельзя ломать без отдельной задачи и проверки.
- `docs/architecture/review.md`
  Срез архитектурных рисков и ограничений.
- `docs/devplan/devplan.md`
  Среднесрочный план развития без подмены task specs.
- `docs/ops/DEPLOYMENT.md`
  Текущий контур запуска, demo/deploy-ограничения и правила production deploy.
- `docs/ops/`
  Эксплуатационные и security-заметки.
- `docs/review/review-log.md`
  Журнал важных проектных изменений и наблюдений.
- `docs/tasks/PROCESS.md`
  Правила ведения задач от task spec до закрытия.
- `docs/tasks/TEMPLATE.md`
  Обязательный шаблон task spec.
- `docs/tasks/*.md`
  Конкретные задачи с нумерацией `001-...md`, `002-...md` и далее.
- `docs/event-classification.md`
  Отдельный domain document, который уже является source of truth для классификации известных типов событий.

## Что считается ключевыми документами перед любой работой
Перед написанием новой задачи и перед началом реализации обязательно перечитывать:
- `docs/WORKFLOW.md`
- `docs/constitution.md`
- `docs/architecture/architecture.md`
- `docs/architecture/invariants.md`
- `docs/devplan/devplan.md`
- релевантные `docs/tasks/*.md`
- релевантные ADR, если они появятся
- `docs/ops/DEPLOYMENT.md`, если задача затрагивает production, deploy, Docker, инфраструктуру, конфиги запуска или эксплуатацию

Это обязательное правило. Если контекст не перечитан, задача считается начатой с нарушением процесса.

## Обязательный процесс работы
1. Сначала изучить текущее состояние репозитория и сопоставить его с ключевыми документами.
2. Если нужна новая работа, сначала создать или обновить task spec в `docs/tasks/`.
3. Зафиксировать в task spec:
   - что именно меняется;
   - чего задача не делает;
   - acceptance criteria;
   - verification;
   - rollback / safety.
4. Только после task spec переходить к реализации.
5. После реализации выполнить verification, указанную в задаче.
6. Обновить затронутые документы:
   - `docs/architecture/architecture.md`, если изменилась фактическая архитектура;
   - `docs/architecture/invariants.md`, если изменились инварианты системы;
   - `docs/architecture/review.md`, если изменился профиль архитектурных рисков;
   - `docs/constitution.md`, если изменились проектные правила;
   - `docs/ops/DEPLOYMENT.md`, если изменился запуск, окружение, infra или deploy;
   - `docs/devplan/devplan.md`, если поменялись приоритеты среднесрочного плана;
   - `docs/review/review-log.md`, если нужно зафиксировать сквозное изменение или новый риск;
   - связанный task spec, указав фактический статус и заметки.
7. Если остался follow-up, создать новую отдельную задачу, а не прятать продолжение в комментариях или TODO.

## Что нельзя делать
- Нельзя начинать реализацию фичи без task spec.
- Нельзя описывать в `docs/architecture/architecture.md` желаемое поведение как уже существующее.
- Нельзя оставлять незадокументированные временные хаки как "нормальное" состояние проекта.
- Нельзя считать README достаточной заменой архитектурной или task-документации.
- Нельзя деплоить в production из "текущего рабочего дерева".

## Правило для production deploy
Любой production deploy допустим только от зафиксированного commit:
- должен быть известен точный commit SHA;
- конфигурация и секреты должны соответствовать именно этому commit;
- verification должна быть выполнена для этого состояния;
- rollback должен ссылаться на предыдущий известный commit, а не на "что было в контейнере раньше".

Если production-контур не описан или сервер/окружение не идентифицированы, это надо явно считать ограничением, а не подразумеваемым знанием.
