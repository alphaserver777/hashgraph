## Контекст
Текущий репозиторий реализует Python-прототип MDRJ-DAG с локальным SQLite-хранением, HTTP API, demo-кластером в Docker Compose и визуализацией `/viz`. В проекте уже есть каталог demo-событий и правила классов `A/B/C`, но нет production-контура для реального получения Linux-сигналов из системных журналов.

Сейчас события создаются приложением напрямую или через demo-сценарии. Для реального внедрения нужен отдельный ingestion-layer, который:
- читает системные журналы Linux;
- минимально нормализует события;
- определяет, относится ли событие к административному доступу или изменению привилегий;
- публикует событие в локальный узел MDRJ-DAG;
- не подменяет собой источник истины и не превращается в отдельную SIEM.

Первая проектная версия такого контура должна быть описана как контейнерный sidecar-агент рядом с локальным узлом реестра. Эта задача фиксирует именно архитектурный и task-level дизайн. Реализация агента как продукта в неё не входит.

## Цель
Зафиксировать decision-complete архитектуру первой реальной production-версии для контейнерного Linux-агента, который собирает минимальный набор событий доступа и привилегий и отправляет их в локальный узел MDRJ-DAG.

## Scope
- Зафиксировать production-модель первой реальной среды: небольшая Linux-инфраструктура.
- Зафиксировать, что агент работает как Docker sidecar рядом с локальным узлом реестра.
- Описать минимальный production-каталог событий классов `A` и `B`.
- Описать минимальный нормализованный контракт события для агента.
- Описать будущий Docker-контур агента.
- Описать правила определения административного субъекта.
- Синхронизировать архитектурные и эксплуатационные документы с этим проектным решением.

## Ограничения
- Не писать код агента, парсеров логов или новых runtime-сервисов.
- Не объявлять production-контур уже реализованным.
- Не подменять текущий demo-каталог событий production-каталогом без явного разведения.
- Не пытаться в этой задаче покрыть Windows, сетевое оборудование, внешние SIEM и сложную межсистемную корреляцию.
- Не проектировать полноценную распределённую модель инцидентов между узлами.
- Не размывать задачу до “сразу построить весь ingestion-pipeline”.

## Текущее состояние
- Узлы MDRJ-DAG работают как Python-процессы с локальным SQLite.
- Docker Compose поднимает только сами узлы `node1/node2/node3`.
- Реальные системные журналы хоста сейчас не читаются.
- Текущий `docs/event-classification.md` описывает demo-каталог:
  - `virus`
  - `mac_spoof`
  - `admin_login`
  - `portscan`
  - `heartbeat`
- В runtime нет production-понятия Linux-агента, который читает `journald`, `auth.log` и `sudo`.

## Предлагаемое изменение

### 1. Production-модель первой реальной среды
- Первой целевой средой считается небольшая Linux-инфраструктура.
- Агент ориентирован на Linux-хосты, где журналирование событий доступа уже ведётся системными средствами.
- Первая цель: доступ и привилегии.
- В первую проектную версию не входят:
  - Windows Event Log;
  - сетевое железо;
  - внешние SIEM-коннекторы;
  - облачные аудиты;
  - сложные правила корреляции между разными платформами.

### 2. Контур контейнерного агента
- На каждом хосте предполагается локальный узел MDRJ-DAG и отдельный sidecar-контейнер агента.
- Агент живёт на том же хосте, что и локальный узел, и отправляет события только в этот локальный узел.
- Агент не является хранилищем истины:
  - источником факта остаётся системный журнал хоста;
  - источником истины для распределённого реестра остаётся SQLite узла MDRJ-DAG.
- Агент не публикует внешний публичный API в первой версии.
- Агент работает как ingestion-layer:
  - читает системные источники;
  - определяет, относится ли событие к нужному production-каталогу;
  - присваивает `event_kind`;
  - присваивает класс `A` или `B`;
  - отправляет событие в локальный HTTP API узла реестра.

### 3. Источники сигналов первой версии
- `journald`
- `auth.log` или его системный эквивалент
- события `sudo`

Если на конкретном Linux-дистрибутиве один из этих источников отсутствует как отдельный файл, это не считается отдельной архитектурой: агент всё равно должен ориентироваться на те же доменные сигналы, даже если читает их из другого системного источника.

### 4. Что идёт в общий распределённый реестр
- В первый production-контур общего реестра входят классы `A` и `B`.
- `A` — подтверждённые привилегированные действия и опасные изменения доступа.
- `B` — неуспешные попытки и значимый контекст расследования.
- `C` в этом production-срезе не является обязательным внешним ingestion-классом.

### 5. Минимальный production-каталог событий

#### Класс A
- `admin_ssh_login_success`
  Успешный административный вход по SSH.
- `sudo_success`
  Успешное получение привилегий через `sudo`.
- `privileged_account_created`
  Создание новой привилегированной учётной записи.
- `privileged_group_membership_changed`
  Добавление пользователя в привилегированную группу или изменение membership, расширяющее привилегии.
- `authorized_keys_changed_for_privileged_user`
  Изменение `authorized_keys` для привилегированной учётной записи.
- `ssh_access_policy_changed`
  Изменение настроек SSH-доступа, расширяющее или открывающее привилегированный доступ.

#### Класс B
- `admin_ssh_login_failure`
  Неуспешная попытка административного входа по SSH.
- `sudo_failure`
  Неуспешная попытка получить привилегии через `sudo`.

### 6. Правило для SSH-входа администратора
- Успешный административный SSH-вход → `A`
- Неуспешная попытка административного SSH-входа → `B`

Под “административным субъектом” в первой версии понимается:
- `root`;
- пользователь из явного списка административных пользователей;
- пользователь из привилегированной группы (`sudo`, `wheel` или их конфигурационный эквивалент).

Это правило должно одинаково трактоваться в документах, будущей конфигурации агента и runtime-коде.

### 7. Минимальный нормализованный контракт события
Каждое событие, которое агент публикует в локальный узел MDRJ-DAG, должно иметь минимум такие поля:
- `event_kind`
- `class`
- `host_id`
- `node_id`
- `occurred_at`
- `collected_at`
- `principal`
- `source_ip`
- `target_service`
- `result`
- `privilege_scope`
- `raw_ref`
- `context`

Минимальная семантика полей:
- `event_kind`
  Стабильный идентификатор типа production-события, например `admin_ssh_login_success`.
- `class`
  Класс `A` или `B`, вычисленный по production-политике агента.
- `host_id`
  Идентификатор Linux-хоста, где реально произошло событие.
- `node_id`
  Идентификатор локального узла MDRJ-DAG, в который агент отправляет событие.
- `occurred_at`
  Время исходного события в системном журнале.
- `collected_at`
  Время, когда агент нормализовал событие и подготовил публикацию.
- `principal`
  Учётная запись, связанная с событием.
- `source_ip`
  IP-источник, если он применим и извлекается надёжно.
- `target_service`
  Целевой сервис, например `sshd` или `sudo`.
- `result`
  `success` или `failure`.
- `privilege_scope`
  Уровень привилегии, например `root`, `sudo`, `admin_group`.
- `raw_ref`
  Ссылка на исходную запись журнала или компактный сырой фрагмент, достаточный для аудита и проверки.
- `context`
  Компактный словарь дополнительных полей, не нарушающий стабильность основного контракта.

### 8. Будущий Docker-контур агента
Первая проектная схема должна предполагать:
- новый Docker-сервис агента рядом с каждым узлом;
- отдельный конфиг агента через YAML или env;
- монтирование системных логов хоста в контейнер агента;
- монтирование production-конфигурации правил классификации;
- сетевое взаимодействие агента только с локальным узлом реестра.

В этой задаче Docker-контур фиксируется на уровне архитектуры, но не реализуется как рабочий compose-сервис.

### 9. Разведение demo и production
- Текущий `docs/event-classification.md` и `mdrj/event_catalog.py` описывают demo-уровень проекта.
- Production-каталог Linux-событий из этой задачи нельзя считать уже введённым в runtime.
- В следующей реализации потребуется либо:
  - развести demo-каталог и production-каталог на уровне документов и кода;
  - либо ввести отдельный production reference document до изменения runtime-классификации.
