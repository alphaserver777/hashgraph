## Контекст
Текущий `/viz` уже помогает наблюдать события, граф, тревоги и аналитические метрики. Но у оператора всё ещё нет отдельного рабочего контура для сопровождения инцидента: негде собрать карточку, заполнить поля, поменять статус и переключиться между списком и доской.

## Цель
Добавить отдельную вкладку `Обработка инцидентов`, где оператор сможет работать с инцидентом как с рабочей сущностью: видеть таблицу, открывать карточку и переключаться на kanban-доску по статусам.

## Scope
- Добавить в UI отдельную вкладку или раздел `Обработка инцидентов`.
- Сделать два режима отображения:
  - таблица;
  - kanban.
- Показать и редактировать базовые поля карточки инцидента.
- Дать возможность менять статус инцидента.
- Показать связанные события внутри карточки или рядом с ней.
- В первой версии ограничить incident-workbench одним операторским узлом, а не всеми нодами кластера.
- Сделать UX карточки ближе к Trello-подобному рабочему сценарию: открыть карточку, редактировать поля, переносить между колонками.
- Явно зафиксировать, является ли первая версия:
  - UI-слоем поверх текущих событий и тревог;
  - или уже отдельной persisted-сущностью.

## Ограничения
- Не называть первую версию полноценной системой incident management, если это только UI-слой.
- Не вводить скрытую новую доменную сущность без явной фиксации в коде и документации.
- Не ломать текущий мониторинговый экран `/viz`.
- Не создавать ложное впечатление, что статус инцидента уже синхронизируется между всеми узлами, если это ещё не реализовано.
- Не заявлять полноценную систему прав доступа, если в первой версии нет backend-проверки ролей и разрешений.
- Если потребуется хранение инцидентов, это должно быть явно зафиксировано как часть задачи, а не спрятано в фронтенде.

## Текущее состояние
- Есть мониторинговая панель, граф, тревоги и аналитика.
- Нет отдельной вкладки обработки инцидентов.
- Нет табличного представления инцидентов.
- Нет kanban-доски по статусам.
- Нет явной карточки инцидента с полями оператора.
- Нет отдельной распределённой модели хранения и синхронизации инцидентов между узлами.
- Нет Trello-подобного сценария открытия карточки, редактирования описания, меток, дат и перемещения по колонкам.

## Предлагаемое изменение
- Добавить новый операторский раздел `Обработка инцидентов`.
- Первую версию вкладки включать только на одном выбранном операторском узле, например `node1`.
- В табличном режиме показывать минимум такие поля:
  - класс события с цветовой индикацией;
  - ID события;
  - тип события;
  - дата и время;
  - название инцидента;
  - статус;
  - приоритет;
  - источник или узел;
  - ответственный;
  - метки;
  - примечание.
- В режиме kanban показывать карточки по стадиям обработки, например:
  - `Новые сигналы`
  - `Проверка`
  - `Подтверждённый инцидент`
  - `Сдерживание`
  - `Восстановление`
  - `Закрыто`
- Дать возможность открыть карточку инцидента в отдельной панели или модальном окне и редактировать в ней минимум:
  - название;
  - описание;
  - примечание;
  - статус;
  - приоритет;
  - ответственного;
  - метки;
  - даты;
  - чек-лист или список шагов;
  - связанные события.
- В карточке отдельно показывать подробности исходного события:
  - класс;
  - ID;
  - тип;
  - время;
  - тело события или его содержательное описание.
- Для kanban-представления поддержать перенос карточки между колонками.
- Для первой версии допустить, что инциденты строятся из событий и тревог, если отдельная persisted-модель пока не вводится.
- Остальные узлы в первой версии продолжают показывать мониторинг, но без incident-workbench.
- Поле `ответственный`, `наблюдатели` и подобные operator fields допустимы, но полноценная система backend-прав доступа не входит в первую версию.
- Если появится хранение инцидентов, отдельно зафиксировать минимальный контракт данных и API.

## Acceptance Criteria
- [ ] В интерфейсе есть отдельная вкладка или раздел для обработки инцидентов.
- [ ] Инциденты видны в табличном представлении.
- [ ] Инциденты видны в kanban-представлении.
- [ ] У карточки инцидента есть понятные поля для оператора.
- [ ] Таблица показывает колонки в операторском порядке: класс события, ID события, тип события, дата/время, инцидент, статус, приоритет, источник, ответственный, метки, примечание.
- [ ] Статус инцидента можно менять.
- [ ] Связь карточки с событиями видна в интерфейсе.
- [ ] Первая версия incident-workbench доступна только на одном операторском узле и это явно понятно из реализации и документации.
- [ ] Карточку инцидента можно открыть и отредактировать в Trello-подобном сценарии.
- [ ] Карточку можно переносить между колонками kanban-доски.
- [ ] В карточке есть минимум описание, метки, даты, ответственный и связанные события.
- [ ] При открытии карточки видны подробности исходного события, а не только общие поля инцидента.
- [ ] Первая версия не притворяется полноценной системой backend-прав доступа, если такой логики нет.
- [ ] Интерфейс явно показывает, demo-слой это или отдельная persisted-сущность.
- [ ] Новая вкладка не ломает текущую мониторинговую панель.

## Rollback / Safety
Если первая версия ограничена UI-слоем, rollback возможен обычным `git revert`. Если добавляется новая persisted-сущность инцидента, rollback должен отдельно учитывать совместимость данных и быть описан до реализации.
