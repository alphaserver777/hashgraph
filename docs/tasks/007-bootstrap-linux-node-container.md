## Контекст
Задача `006-self-contained-linux-node-container.md` зафиксировала целевую production-модель: один Docker-контейнер на Linux-хосте представляет один узел MDRJ-DAG и внутри себя объединяет ingestion-контур, runtime реестра, локальную SQLite-базу и встроенный UI. Но в коде проекта пока нет ни каркаса встроенного ingestion-layer, ни конфигурации узла, которая явно включает реальный Linux-контур событий доступа и привилегий.

Следующий шаг не должен сразу превращаться в полный production-агент. Нужен узкий вертикальный срез, который:
- закладывает форму самодостаточного контейнера узла;
- вводит минимальный ingestion-контур внутри узла;
- позволяет поднять контейнер и обработать минимум одно реальное Linux-событие;
- не подменяет собой полный каталог production-сигналов и сложную корреляцию.

## Цель
Подготовить минимальный рабочий каркас самодостаточного Linux-контейнера узла, который может принять хотя бы один реальный тип Linux-сигнала из контура доступа и привилегий и записать его в локальный распределённый реестр.

## Scope
- Добавить минимальный ingestion-контур внутрь контейнера узла.
- Зафиксировать отдельную конфигурацию для Linux-режима узла.
- Подготовить Docker-контур и compose-конфигурацию для такого узла.
- Выбрать один минимальный реальный сигнал для первого вертикального среза.
- Обеспечить попадание этого сигнала в локальный DAG, UI и хранилище узла.
- Обновить документы по факту введения каркаса.

## Ограничения
- Не пытаться в этой задаче покрыть весь production-каталог из `006`.
- Не строить полноценный универсальный парсер `journald`, `auth.log` и `sudo` сразу.
- Не добавлять отдельный внешний сервис-агент или sidecar.
- Не заявлять готовность production deployment.
- Не смешивать эту задачу с Windows, сетевыми устройствами, внешним SIEM и multi-tenant логикой.
- Не реализовывать полноценную схему корреляции инцидентов.

## Текущее состояние
- Контейнер узла уже существует как Python runtime с HTTP API, UI и SQLite.
- Dockerfile и `docker-compose.yaml` поднимают только demo-узлы.
- Реальные системные журналы хоста пока не читаются.
- Контур событий доступа и привилегий описан на уровне проектной модели в `006`, но не реализован в runtime.

## Предлагаемое изменение

### 1. Минимальный вертикальный срез
Первая реализуемая версия должна брать только один реальный сигнал:
- `admin_ssh_login_success`

Это событие выбрано как базовый production-сигнал, потому что:
- оно уже согласовано как `A`;
- оно достаточно понятно оператору;
- оно позволяет проверить весь путь от ingestion до DAG, UI и хранения;
- оно не требует сразу покрыть все типы Linux-событий.

### 2. Каркас встроенного ingestion-контура
Внутри узла должен появиться минимальный ingestion-layer, который:
- включается отдельной конфигурацией;
- умеет читать один заданный Linux-источник;
- выделяет успешный административный SSH-вход;
- нормализует событие в production-формат;
- передаёт его во внутренний runtime узла.

В первой версии допустимо опереться только на один источник данных, если это явно задокументировано и не ломает дальнейшее расширение.

### 3. Минимальный контракт первого сигнала
Для `admin_ssh_login_success` должны реально проходить в runtime минимум поля:
- `event_kind`
- `class`
- `host_id`
- `node_id`
- `occurred_at`
- `collected_at`
- `principal`
- `source_ip`
- `target_service`
- `result`
- `privilege_scope`
- `raw_ref`
- `context`

### 4. Конфигурация Linux-режима узла
Нужен отдельный конфигурационный контур для Linux ingestion, который в первой версии задаёт:
- включён ли встроенный ingestion;
- какой источник используется;
- список административных пользователей;
- список привилегированных групп;
- локальный идентификатор хоста;
- политику первого сигнала `admin_ssh_login_success`.

### 5. Docker-контур
Нужно подготовить Docker-контур узла для Linux-режима:
- один контейнер = один узел;
- монтирование нужного системного источника событий хоста;
- монтирование production-конфига ingestion;
- явный способ запуска такого узла отдельно от demo-кластера.

В первой версии это может быть отдельный compose-файл, отдельный профиль или отдельный сервис, но в задаче и коде это должно быть оформлено однозначно.

Фактически выбранный вариант для первого среза:
- отдельный сервис `linux-node` в `docker-compose.yaml`;
- отдельный профиль `linux-node`;
- отдельный универсальный конфиг `docker/configs/linux-node.yaml`;
- входной источник для первой версии: файл `auth.log`, примонтированный в контейнер как read-only fixture.

### 6. Видимость в UI
Минимальный сигнал должен быть виден:
- в графе событий;
- в классификации `A/B`;
- в инцидентной панели, если инциденты включены на операторском узле;
- без визуальной подмены demo-событиями.

### 7. Разведение demo и Linux-режима
Нужно явно развести:
- demo-контур текущего прототипа;
- Linux-режим с реальным ingestion-сигналом.

Нельзя, чтобы после задачи было непонятно, какие события синтетические, а какие пришли из реального Linux-источника.

## Rollback / Safety
Изменение должно быть обратимо через `git revert`. Если новый Linux-режим ломает demo-контур, rollback должен возвращать demo-запуск без дополнительных ручных шагов. Нельзя заменять текущий demo-контур новым Linux-режимом без явного разделения.
