## Контекст
Сейчас состав сети задаётся в основном через YAML-конфиг и API `peers/register`. Такой контур подходит для demo и первого внешнего стенда, но неудобен для оператора: нет единой страницы, где видны все участники, нет явного статуса `включён/исключён`, и нет простого способа исключить узел после компрометации без ручного редактирования конфигов.

Одновременно для текущей модели доверия не нужен сложный zero-trust discovery: предполагается, что участники сети изначально доверены, но любой из них может быть скомпрометирован и затем должен быть быстро исключён из gossip-контура.

## Цель
Добавить в веб-панель operator-managed страницу участников сети, где можно просматривать, добавлять, временно исключать и удалять участников из локального peer-реестра узла.

## Scope
- Добавить новую страницу/раздел `Участники сети` в `/viz`.
- Показать сохранённый локальный реестр участников сети.
- Дать оператору возможность:
  - добавить новый узел;
  - сохранить заметку по узлу;
  - исключить узел из активного gossip-контура;
  - вернуть ранее исключённый узел;
  - удалить узел из локального реестра.
- Перенести peer-registry из purely runtime-представления в сохраняемое состояние SQLite.
- Развести:
  - активных участников gossip;
  - сохранённые, но исключённые участники.

## Ограничения
- Не строить в этой задаче полноценный автодискавери.
- Не вводить PKI, сертификаты, zero-trust handshake или централизованный registry service.
- Не утверждать, что локальное исключение узла уже автоматически распространяется на весь кластер.
- Не смешивать эту задачу с incident-management или Linux ingestion beyond текущий срез.

## Текущее состояние
- `NodeConfig.peers` задаёт начальный список пиров.
- `Node.register_peer()` умеет только добавить адрес в runtime и SQLite.
- В `/viz` нет отдельной страницы для управления участниками сети.
- Таблица `peers` в SQLite существует, но используется как вспомогательное хранение last_seen/healthy и не даёт operator semantics `enabled/excluded`.

## Предлагаемое изменение
- Расширить peer-registry узла до локально сохраняемой operator-модели:
  - `address`
  - `last_seen`
  - `healthy`
  - `enabled`
  - `note`
  - `source`
- Считать активными gossip-peer только те узлы, у которых `enabled = true`.
- Добавить в `/viz` отдельный раздел `Участники сети` с таблицей и формой добавления.
- Добавить backend endpoints для:
  - чтения peer-registry;
  - обновления `enabled` и `note`;
  - удаления участника.
- Конфиг-пиры при первом bootstrap должны попадать в локальный peer-registry, но не обязаны переопределять operator-исключение узла после рестарта.

## Rollback / Safety
Изменение должно откатываться обычным `git revert`. При rollback узел должен вернуться к старому поведению со списком пиров из конфига и простым `/peers/register`, без необходимости ручного восстановления данных событий.

## Заметки
- Эта задача сознательно не делает discovery “по недоверию”. Модель текущего этапа: узлы изначально доверены, но могут быть оперативно исключены после компрометации.
- Следующий возможный шаг после этой задачи: seed-based обнаружение узлов с operator-подтверждением кандидатов, а не полный ручной ввод всех адресов.
