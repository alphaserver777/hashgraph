# Инварианты Системы

## Инварианты Графа
- Сохранённый граф должен оставаться ацикличным. `DAGStorage.toposort()` выбрасывает ошибку при цикле; код не должен обходить это предположение.
- Каждое не-genesis событие по возможности должно ссылаться максимум на двух родителей и стараться иметь не меньше двух ссылок, если доступны self-parent, удалённый кандидат и/или anchors.
- Связи parent-child выводятся из списка родителей события и должны оставаться синхронизированными с фактом сохранения события.

## Инварианты Репликации
- Идентификаторы событий неизменяемы и дедуплицируются первичным ключом.
- Повторный ingest уже известного события может объединять `path_meta`, но не должен создавать дубликаты строк события или менять содержимое payload.
- Gossip обязан ставить вновь сохранённые события в очередь на дальнейшую репликацию.

## Инварианты Упорядочивания
- `consensus_ts` должен вычисляться детерминированно из локально наблюдаемого состояния события и envelope, после чего итоговый порядок разрешается через tie-break по `event.id`.
- Если меняется семантика порядка, интеграционные тесты должны подтверждать сходимость между узлами после снятия partition.

## Инварианты Хранения
- SQLite является источником истины для событий, связей графа, envelope и сохранённого состояния пиров.
- В пустом хранилище genesis anchors должны появиться до начала обычной эмиссии событий.
- Genesis anchors могут содержать identity-метаданные известных узлов, но должны оставаться обычными `C`-событиями без родителей.
- Запрос frontier должен отражать листовые вершины в соответствии с таблицей `edges`.

## Инварианты Интерфейсов
- CLI-команды должны оставаться тонкими обёртками над HTTP endpoint или helper-логикой жизненного цикла узла.
- API payload должны оставаться canonical JSON object/array, как задокументировано в коде и пользовательской документации.
- Конфиги должны оставаться YAML-ориентированными и корректно маппиться на `NodeConfig`.

## Инварианты Документации
- Любое изменение формы события, формы endpoint, жизненного цикла узла или сценария развертывания должно отражаться в `docs/`.
- Перед новой задачей и перед началом реализации нужно перечитывать ключевые документы, перечисленные в `docs/WORKFLOW.md`.
- Если остаётся технический долг или незавершённое продолжение, оно должно фиксироваться отдельной задачей, а не теряться в комментариях.
