# ADR-0002: HMAC-SHA256 аутентификация state-changing endpoints HTTP API

## Статус
Принято (этап 0). Транзитное решение — должно быть пересмотрено в этапе 3 в пользу Ed25519 per-node keypair, см. секцию «См. также».

## Контекст
До этого решения HTTP API узла принимал любые POST/PUT-запросы без проверки отправителя:
- `POST /event/emit` — клиент мог писать любые события в распределённый реестр.
- `POST /event/batch` — gossip-эндпоинт принимал envelope-ы без проверки источника.
- `POST /peers/*` — администрирование peer-registry было открытым.
- `POST /viz/clear`, `POST /viz/simulation/control` — деструктивные операции пропагировались между узлами без подписи.

В контексте задачи диссертации (защита от УБИ.124 — подделка журнала регистрации событий) открытый API делает прототип уязвимым к простейшей атаке: атакующий с сетевым доступом подделывает события, минуя коллектор, и tamper-proof свойство DAG не помогает, потому что событие легитимно подписано добавляющим узлом.

Существующий `security.hmac_key` в конфиге использовался только для опциональной подписи payload отдельных событий — не для аутентификации транспорта.

## Решение
Введён aiohttp middleware `hmac_auth_middleware` в `mdrj/api.py`, который для методов POST/PUT/DELETE/PATCH:

1. Если `security.hmac_key` **не задан** — пропускает запрос (этап разработки, prototype-режим). При старте `build_app` пишется warning в лог.
2. Если ключ задан — требует заголовок `X-MDRJ-Sig` с hex-encoded HMAC-SHA256 от сырых байтов тела запроса. Сравнение через `hmac.compare_digest` (constant-time).
3. При расхождении возвращается 401 с записью в лог `mdrj.api.auth`.

GET-эндпоинты (`/status`, `/metrics`, `/dag`, `/viz`, `/peers`, `/incidents`, `/consensus/digest`, `/viz/stream`) не защищаются — это сознательное решение для упрощения мониторинга. Защита GET-эндпоинтов перенесена в задачу этапа 5 (web UI auth с session cookie + role gating).

Симметрично с серверной стороной все межузловые POST-вызовы теперь подписывают тело:
- `mdrj/gossip.py:GossipEngine._send_to_peer` — принимает `hmac_key` в конструкторе.
- `mdrj/node.py:_send_simulation_request` и `_send_clear_request` — используют конфиг узла.
- Helper `mdrj/utils.py:signed_request_body(payload, hmac_key)` инкапсулирует логику.

Подпись считается над **сырыми байтами тела запроса**, а не над canonical JSON. Это упрощает middleware (не надо парсить и пере-сериализовать body), и достаточно для аутентификации transport, поскольку байты — это то, что атакующий мог бы подменить in-flight.

## Последствия
**Положительные:**
- При корректной конфигурации (`security.hmac_key` одинаков на всех узлах) внешний клиент без ключа физически не может писать в реестр, в т.ч. через gossip.
- Это закрывает один из вектор-атак на УБИ.124 — подделку записей через прямой POST в обход коллектора.
- Подготавливает почву для этапа 1: внешние коллекторы тоже должны будут подписывать свои POST `/event/emit`.

**Отрицательные / ограничения:**
- **Общий симметричный ключ.** Все узлы и коллекторы используют один и тот же `hmac_key`. Утечка ключа на одном узле компрометирует transport-уровень всей сети. Это приемлемо для прототипа, но не для production.
- **Нет replay-защиты.** Подписанный запрос можно повторно отправить (`X-MDRJ-Sig` валиден до тех пор пока валиден ключ). Для критичных эндпоинтов это уже частично смягчено идемпотентностью (`event/batch` дедуплицирует по event_id, `viz/clear` использует token-based дедупликацию). Для production нужен timestamp + nonce.
- **GET-эндпоинты открыты.** Чтение состояния возможно без аутентификации. Это сознательно — закрывается в этапе 5 через web UI auth.
- **Конфликт миграции.** Если оператор включает hmac_key на одном узле, а на других ещё нет — gossip между ними сломается (узлы без ключа не подпишут запросы и получат 401). Включать ключ нужно одновременно во всём кластере, желательно с rolling restart.

**Что НЕ делает это решение:**
- Не вводит per-node identity (Ed25519 keypair) — это запланировано в этапе 3 как часть подписи checkpoint.
- Не аутентифицирует пользователей UI (login/password) — это этап 5.
- Не защищает gossip от атак с уже скомпрометированных узлов сети.
