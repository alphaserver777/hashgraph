# ADR-0006: Retention policy, event_skeleton GC и холодный архив

## Статус
Принято (этап 3.b прототипа). Этот ADR опирается на ADR-0005 (checkpoint механизм) и завершает диссертационный этап ротации реестра.

## Контекст
ADR-0005 ввёл подтверждённые подписями ≥2/3 узлов **checkpoint**-снэпшоты состояния реестра до `round_received=N`. Без сопутствующей политики удаления это лишь маркер — горячий реестр всё равно растёт линейно. ADR-0006 описывает, **что именно** мы делаем со старыми событиями после подтверждённого checkpoint, и как сохраняется криптографическая проверяемость прошлого после ротации.

Старый `gc_by_quota` сносит события вместе с edges без всякой защиты. Это противоречит цели диссертации — он удаляет улики и причинную цепь, которая нужна для расследования инцидента ИБ.

## Решение

### Конфигурация (`mdrj/config.py:RetentionConfig`)
- `enabled` — по умолчанию `false` (опт-ин для прототипа).
- `max_db_bytes` — будущий мягкий триггер; в текущей реализации не используется (`max_age_days` достаточно для прототипа).
- `max_age_days` — событие должно быть **старше** этого порога, чтобы стать кандидатом на prune. Время отсчитывается от `created_at` события, не `ts_local`. Это даёт оператору окно для ручной проверки горячих инцидентов.
- `keep_class_a` — критичные события (класс A) **никогда** не удаляются автоматически. Это исходит из требования диссертации: события подозреваемого инцидента всегда остаются доступными в полном объёме.
- `archive_path` — опциональный JSONL-файл для cold archive. Если пуст, события удаляются без архива (только skeleton).
- `poll_interval_sec` — частота фонового retention loop (по умолчанию 300 с = 5 минут).

### Процесс retention (`Node.run_retention_once`)
1. Если `retention.enabled = false` → no-op.
2. Получить последний `confirmed` checkpoint из `storage.latest_confirmed_checkpoint()`.
3. Если confirmed checkpoint нет → no-op. Без подтверждённого 2/3-кворума мы **не имеем права** удалять события.
4. Вызвать `storage.prune_under_checkpoint`:
   - WHERE `round_received IS NOT NULL AND round_received <= checkpoint_round`
   - AND `created_at < now - max_age_days*86400`
   - AND (если keep_class_a) `cls != 'A'`
5. Для каждого подходящего события **до удаления**:
   - Вычислить `payload_hash = SHA256(payload_text)`.
   - Записать `event_skeleton(id, cls, parent_ids, round_received, payload_hash, archived_at)`.
   - Если есть `archive_path` — дописать сериализованную полную запись в JSONL.
6. Удалить event из `events` и `envelopes`. **Edges намеренно НЕ удаляются** — оставшиеся события-потомки продолжают ссылаться на родителей через event_skeletons.

### Фоновый loop
- `_retention_loop` в Node запускается при `retention.enabled = true` в `start()` и останавливается в `stop()` через `_retention_stop` event.
- Цикл: `await asyncio.to_thread(self.run_retention_once)`, затем `asyncio.wait_for(stop.wait(), timeout=poll_interval_sec)`.

### Холодный архив (формат JSONL)
Один header в начале chunk + один event на строку:
```jsonl
{"_archive_header": {"checkpoint_round": 100, "merkle_root": "abc…", "members_snapshot_hash": "def…", "exported_at": 1779…}}
{"id": "…", "cls": "B", "source": "…", "creator": "…", "ts_local": …, "vclock": {…}, "parents": [...], "payload": {…}, "consensus_ts": …, "round_received": 99, "payload_hash": "…"}
{"id": "…", …}
```
Несколько вызовов retention аппендят новые header+chunks в тот же файл.

### CLI команды
- `mdrj checkpoint propose [--round-received N] --config …` — POST на `/checkpoint/propose`.
- `mdrj checkpoint list [--status confirmed] [--limit N] --config …`.
- `mdrj checkpoint verify --round-received N --config …` — exit code 2 если `has_tamper_evidence: true`.
- `mdrj archive verify <path>` — оффлайн-проверка: для каждой записи перевычисляет SHA256 payload и сравнивает с записанным `payload_hash`. Exit code 2 при любом mismatch.

## Последствия

**Положительные:**
- **Размер горячего хранилища ограничен**, без потери criminal-record-уровня проверяемости. Скелет события занимает ~100 байт против килобайтов payload.
- **Causal chain проверяется** даже после prune: edges + event_skeletons → можно отследить родительские ссылки от любого живого события вплоть до genesis. Это важно для расследования: «откуда взялось это событие».
- **Tamper evidence сохраняется** через payload_hash в скелете. Если у атакующего был доступ к узлу и он модифицировал старый event ДО pruning — нашиMerkle-roots в checkpoint его поймают; если ПОСЛЕ pruning — payload уже физически нет, но архив или другой узел даёт ground truth.
- **Cold archive проверяется оффлайн** через `mdrj archive verify` — оператор может перенести JSONL на изолированный носитель и периодически валидировать без сетевого доступа к узлу. Это даёт «вечное» свидетельство.
- **Class A события защищены** дефолтным `keep_class_a=true` — самые критичные сигналы ИБ остаются полностью видимыми.

**Отрицательные / ограничения:**
- **`max_db_bytes` в текущей реализации не enforced.** Используется только `max_age_days` + `keep_class_a`. Размер БД остаётся производной функцией от потока событий и `max_age_days`. Для production нужен жёсткий cap через приоритетное удаление самых старых eligible events до достижения порога.
- **Архив не подписан целиком.** Каждое event в архиве имеет HMAC sig (если оператор это включил per-event), но сам файл архива не имеет общей подписи. `archive verify` проверяет только payload_hash. Тип атаки «полная замена архивного файла» не покрыт — нужна доп. подпись archive header чейном.
- **Edges не удаляются.** Это сознательное решение для сохранения causal chain, но означает, что таблица edges растёт даже когда events удаляется. Долгосрочно нужна edge-pruning стратегия (например, удаление edges, у которых **оба** конца — skeletons).
- **`run_retention_once` синхронный.** Запускается через `asyncio.to_thread`. При тысячах удаляемых событий блокирует один поток на несколько секунд. Допустимо для прототипа, не для production.
- **Skeleton не реплицируется** между узлами. Если узел потерял данные и присоединился к сети — он получит только события, которые ещё живы в горячем хранилище peers. Восстановление полной истории невозможно. Это **по дизайну** — узел должен получать checkpoint-снэпшоты от соседей и догонять с этой точки.
- **Архив можно читать и подделать на одном узле.** Реальная защита от подмены архива — несколько узлов с независимыми архивами + периодическая verify. Это операционная мера, не свойство кода.

## Связь с диссертационной моделью
Запись `bytes_per_event` (из ADR-0004) умножается на сохраняющиеся события класса A и складывается со средним размером skeleton (~150 байт) на удалённые B/C. Это даёт прогнозируемый верхний край `K_рес` (затрат ресурсов) для архитектуры A4 в `simulation.html` — теперь его можно эмпирически замерить на 24-часовом прогоне и подставить в модель вместо предположения.
