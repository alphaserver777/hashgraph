# Обзор Архитектуры И Рисков

Дата среза: 2026-04-03

## Высокий Приоритет
- Garbage collection в `DAGStorage.gc_by_quota()` может удалять исторические события и связи без сохранения гарантий замыкания графа для будущей реконструкции. Если ограничение квот станет реальным рабочим механизмом, могут пострадать консенсус и отладка.
- Модель доверия транспорта слабая: gossip использует обычный HTTP без аутентификации пиров и без TLS, а HMAC для payload опционален и не принуждается на входе.
- API и storage contracts ещё не оформлены как отдельные рабочие документы, поэтому при следующем изменении высок риск неверных допущений.

## Средний Приоритет
- HTML/JS визуализатора встроен прямо в `mdrj/api.py`, из-за чего изменения интерфейса тяжелее проверять и отделять от серверной логики.
- Состав пиров статический или ручной, поэтому восстановление кластера и масштабирование зависят от действий оператора.
- `lamport_ts` хранится в базе, но не является центральной частью текущего пайплайна упорядочивания; без отдельной документации это может путать следующих участников.
- Появился первый встроенный Linux ingestion-loop, но он пока file-based и понимает только `admin_ssh_login_success`. Расширение до реального набора `journald/auth.log/sudo` остаётся отдельным риском и задачей.

## Низкий Приоритет
- В репозитории лежат закоммиченные `.db`-файлы в `data/`. Для demo это удобно, но размывает границу между fixtures и живым состоянием.
- В проекте смешиваются русские и английские строки интерфейса и документации. Это не блокер, но затрудняет поддержание единого стиля.
- Runtime-каталог и demo-сценарии теперь разведены, но UI и документация всё ещё легко перепутать, если не поддерживать это разделение дисциплинированно.
- Peer-registry теперь можно редактировать из UI, но это локальное решение конкретного узла. Без отдельной политики синхронизации и доверенного discovery разные узлы могут видеть разный состав участников.

## Вывод
- Текущая архитектура согласована для уровня прототипа и достаточно структурирована для итеративного развития.
- Ближайший полезный шаг, это формализация API и storage contracts, не теряя при этом узкий фокус на следующем расширении Linux ingestion после первого рабочего сигнала.
