# MDRJ-DAG: текст для буклета

## MDRJ-DAG: распределенный журнал событий безопасности с DAG-репликацией, gossip-обменом и детерминированным упорядочиванием

## Краткое описание
Программное обеспечение предназначено для сбора, распространения и согласования событий информационной безопасности в распределенной среде. Система формирует направленный ациклический граф событий, обеспечивает репликацию между узлами без центрального координатора и восстанавливает единый детерминированный порядок обработки инцидентов. Актуальность решения связана в том числе с тем, что одной из задач атакующего по MITRE ATT&CK T1070 является уничтожение или подделка логов, включая журналы на центральном сервере, чтобы сорвать расследование и атрибуцию инцидента.

## Назначение программы
Программа используется для повышения отказоустойчивости распределенного мониторинга ИБ, синхронизации журналов между узлами и сохранения причинно-следственных связей между событиями. Решение ориентировано на работу в условиях частичной деградации сети, временной недоступности узлов и неоднородной инфраструктуры.

## Основные возможности
- формирование DAG-журнала событий с родительскими связями;
- leaderless gossip-репликация между узлами;
- приоритетная доставка событий классов A/B/C, где в текущем прототипе классы A и B транслируются принудительно, а класс C отправляется по необходимости;
- детерминированное упорядочивание событий на основе локально вычисляемого `consensus_ts`;
- локальное долговременное хранение в SQLite;
- контроль состояния узлов, метрик и связности кластера;
- визуализация графа событий в реальном времени.

## Характеристики программного обеспечения
- язык реализации: Python 3.11+;
- сетевое взаимодействие: HTTP JSON API;
- режим работы: одиночный узел или кластер;
- хранение данных: локальная SQLite-база на каждом узле;
- развертывание: локально или в Docker Compose;
- тестирование: unit и integration tests для gossip, consensus и merge after partition.

## Особенности функционирования
- отсутствие центрального управляющего узла;
- сохранение работоспособности при временном разделении сети;
- повторная синхронизация DAG после восстановления связи;
- дедупликация событий и повторно используемых envelope;
- восстановление общего порядка событий без внешнего консенсусного сервиса.

## Реализованный подход
В основе программы лежит DAG-журнал, в котором новое событие по возможности ссылается на два родителя: локального предшественника и одну из недавних удалённых вершин, а при нехватке кандидатов используются genesis anchors. Репликация выполняется эпидемическим gossip-обменом, а итоговый порядок событий вычисляется по `consensus_ts`, который в текущем прототипе выводится из Lamport-подобного счётчика, deterministic bias по источнику и наблюдаемых hop timestamp.

## Достигаемый результат
Применение программы позволяет:
- повысить устойчивость распределенного сбора событий ИБ;
- сократить риск потери критичных инцидентов при сбоях связи;
- обеспечить согласованное представление журнала на разных узлах;
- сохранить причинно-следственные связи между событиями;
- повысить прозрачность анализа инцидентов и последующей реконструкции их развития.

## Ключевые преимущества
- безлидерная архитектура;
- детерминированный порядок событий;
- приоритетная репликация критичных сообщений;
- простое локальное развертывание;
- наглядная визуализация hashgraph/DAG;
- расширяемость по транспорту, политике безопасности и хранилищу.
